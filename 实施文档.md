# 《基于 MCP 的本地化个人知识库助手》实施文档

## 1. 项目目标与范围

### 1.1 本章输入
- 课题名称：基于 MCP 的本地化个人知识库助手。
- 已批准写作计划：系统定位为小范围、真实可用、可稳定演示的 PC 桌面端个人知识库助手，其中“可稳定演示”的判定规则见 1.3 节。
- 已确认能力范围：PDF、MD、DOCX 文档处理，文本清洗，文本切块，持久化 RAG（将文档切块结果长期保存，并以检索结果增强答案生成的处理链路），检索问答，上下文调用与隔离（在当前回答中引用同一会话或指定历史会话中的相关信息，并阻断未显式选定会话的内容混入当前回答），上下文压缩（将历史对话压缩为保留关键事实、问题和结论的短文本），token 消耗记录与展示接口预留（token 为模型输入输出文本的计量单位），来源文档与文本块追溯，有限学术图像关键信息提取（输出限定为结构化文本、摘要或证据片段）。
- 已确认排除范围：Web、多用户、权限控制、云同步、跨设备协同、移动端、平台化插件生态、大规模网页采集、与论文场景无关的多模态扩展、通用智能体系统。

### 1.2 本章输出
- 项目目标陈述。
- 系统纳入范围说明。
- 系统排除范围说明。
- 后续章节必须遵守的范围约束。

### 1.3 约束
- 系统交付形态固定为 **PC 桌面端应用**。
- 系统目标固定为 **单一用户的知识处理闭环**，不得扩展为面向组织或平台的通用系统。
- MCP（Model Context Protocol，下文指按需连接外部工具或资源的协议/调用机制）固定为 **按需调用的工具或协议层能力**，不得被写成系统的架构中心特征。
- 模型服务允许通过 API（应用程序编程接口）调用，**本地化** 的含义固定为应用运行在本地 PC 环境中，不包含“模型必须本地部署”的附加要求。
- **普通检索与回答** 固定定义为：用户提出单一问题，系统无需比较多个历史问题、无需总结整段会话、无需跨多个步骤重组问题链路即可完成回答。
- **复杂问题** 固定定义为：用户要求总结既有会话、连续追问同一问题、比较多个前序问题，或要求系统跨多个文档片段综合作答。
- 性能计时起点固定为用户提交完整问题且相关文档已经完成导入与索引；计时终点固定为系统在界面输出可阅读结果。
- **初步结论** 固定定义为：输出中已给出当前问题的直接判断或阶段性答案，并至少附带 1 条来源文档线索。
- **最终结论** 固定定义为：输出中已给出完整答案，并同时附带来源文档与文本块追溯信息以及 token 消耗记录。
- 普通检索与回答需在 3 分钟内输出最终结论；复杂问题需在 3 分钟内输出初步结论，并在 5 分钟内输出最终结论。
- **可稳定演示** 固定定义为：在导入文档集合不变的条件下，系统连续处理 20 次复杂问题时不发生进程崩溃、不发生会话内容串扰、每次回答均可给出来源文档与文本块追溯信息以及 token 消耗记录。

### 1.4 本章边界
- 纳入边界：文档导入、文本预处理、持久化 RAG、检索问答、上下文管理、token 消耗记录与展示接口预留、来源文档与文本块追溯、有限学术图像关键信息提取。
- 排除边界：Web 访问入口、多用户权限体系、云同步、跨设备协同、移动端、开放插件生态、大规模网页采集、无关多模态扩展、通用智能体编排。
- 输入形态边界：PDF、MD、DOCX 为主；纯文本仅作为统一文本中间形态或辅助输入，不构成独立输入域。
- 图像能力边界：图像处理对象限定为学术论文中的统计图、表格、公式等内容，输出限定为结构化文本、摘要或证据片段，不输出独立图像分析报告，不承担通用视觉理解任务。

### 1.5 本章排除项
- 不将系统目标扩展为多用户协作平台、组织级知识平台或通用智能体系统。
- 不将 Web 入口、云同步、跨设备协同、移动端和开放插件生态写入本章纳入范围。
- 不将纯文本写成用户独立输入域，不将学术图像处理写成通用视觉理解系统。
- 不将 MCP 写成系统主控中心，不将“本地化”解释为“模型必须本地部署”。

### 1.6 本章验收判定
- 本章必须同时出现 **系统交付形态**、**系统目标**、**纳入范围**、**排除范围** 四类明确表述。
- 本章必须给出 **普通检索与回答**、**复杂问题**、**初步结论**、**最终结论**、**可稳定演示** 的固定定义。
- 本章必须给出性能计时起点和计时终点，且时间要求写为可检查的数值阈值。
- 本章必须明确写出 PDF、MD、DOCX 为主要输入，纯文本不构成独立输入域。
- 本章不得出现 Web、多用户、云同步、移动端、开放插件生态被纳入范围的表述。

### 1.7 正文
本项目的目标是构建一个面向单一用户、面向学术文献、面向稳定演示的本地化个人知识库助手。该系统以 PC 桌面端为唯一交付形态，以文档导入、文本清洗、文本切块、持久化 RAG、检索问答、上下文调用与隔离、上下文压缩、token 消耗记录、来源文档与文本块追溯为核心链路，形成从资料接入到答案生成再到证据定位的闭环过程。项目的研究重点不在于展示功能泛化且范围扩张的多功能平台，而在于在严格收敛的范围内实现可用、可验证、可维护的知识处理流程。

在范围定义上，系统只处理 PDF、MD、DOCX 三类主要文档来源，并允许在内部处理流程中将其统一转换为文本中间形态。该文本中间形态仅用于系统内部清洗、切块、索引和回答生成，不作为用户独立导入或编辑的输入类型。系统同时支持对学术论文中统计图、表格、公式等对象进行有限学术图像关键信息提取，但输出仅限结构化文本、摘要或证据片段，该能力不扩展为通用视觉理解能力。所有纳入的功能点均直接服务于“个人知识处理闭环”这一项目目标，不承担平台化扩展职责，不承担协同办公职责，也不承担通用多模态服务职责。

在技术定位上，MCP 的角色被限定为按需调用的外部能力接入机制，其功能价值体现在组织必要工具调用，而不体现在主导业务主流程。由此形成的项目范围具有明确边界：系统关注知识处理主链路，排除 Web 化、组织化、平台化和泛化扩展，保证实施文档的后续章节能够围绕同一目标进行展开，不发生范围漂移。

### 2.7 正文
本系统的直接用户为单一用户，其典型身份为承担论文阅读、资料整理、内容归纳和答辩准备工作的个人研究者。该用户的核心诉求不是访问一个开放知识平台，而是在本地桌面端环境中持续处理个人文献资料，并在文献阅读与问题求解过程中获得稳定的问答支持、会话内容总结支持和证据追溯支持。因此，系统的用户建模不涉及角色分层、权限体系和团队协作逻辑，而是围绕单一用户的连续知识处理活动展开。

从典型场景看，用户首先需要将 PDF、MD、DOCX 文档导入系统，并完成清洗、切块和持久化存储；随后，用户围绕文献内容提出事实性问题、解释性问题和比较性问题，系统根据持久化 RAG 结果生成回答，并回溯到来源文档与文本块。除单轮问答外，系统还需要支持连续追问、对同一问题的进一步澄清、对前文多个问题的比较，以及对既有会话内容的总结。这些场景共同构成了“资料接入—知识提取—问题求解—证据回溯”的完整使用链路。

| 场景 | 判定条件 | 对应系统能力 | 受限边界 |
| --- | --- | --- | --- |
| 事实检索 | 用户提出单一事实性问题，不要求比较历史问题或总结会话 | 检索、答案生成、来源文档与文本块追溯 | 不扩展为开放式联网搜索 |
| 概念解释 | 用户要求说明术语、方法或文献结论含义 | 持久化 RAG、检索增强生成 | 结论仅基于已导入文档 |
| 结果比较 | 用户要求比较多个文档片段、多个前序问题或多个结论 | 上下文调用、上下文隔离、上下文压缩、答案生成 | 不扩展为跨用户协作比较 |
| 连续追问 | 用户围绕同一问题继续追问、澄清或缩小范围 | 上下文调用、上下文隔离、token 记录 | 不允许未显式选定会话内容混入当前回答 |
| 会话内容总结 | 用户要求总结当前会话或指定历史会话内容 | 历史对话聚合、上下文压缩、答案生成 | 仅总结当前会话或用户显式指定的历史会话 |
| 来源文档与文本块追溯 | 用户要求定位回答依据 | 来源文档与文本块追溯、token 记录 | 不扩展为外部知识库审计报告 |

本章所界定的用户与场景只服务于已批准的项目范围，不引入新的产品目标。系统在用户场景中的职责是完成知识处理与问答支持，而不是承担通用办公助手、通用客服系统或通用知识平台的职责。由此可以保证后续章节围绕统一的用户对象和统一的使用环境展开，不因场景扩大而改写项目目标。

### 3.7 正文
本系统的边界以“PC 桌面端应用、知识处理闭环、外部能力按需接入”为判定基准。系统内部承担从文档导入、文本预处理、持久化索引、检索问答、上下文管理到来源文档与文本块追溯的核心责任，这些责任共同构成系统的主流程。系统外部则保留模型 API、MCP 所连接的外部工具或资源以及操作系统文件环境等依赖。边界划分的目的在于明确哪些能力由系统自行完成，哪些能力通过外部接口获得，从而避免在后续设计中出现职责混淆。对于文档对象，导入前的原始文件属于系统外部输入对象，导入并解析后形成的文本块、索引和元数据属于系统内部处理对象。

在边界控制上，任何超出单一用户、超出本地桌面交互、超出文档知识处理闭环的能力均被划入系统边界之外。Web 入口、多用户权限、云同步、跨设备协同、移动端和平台化插件生态均不进入本系统范围。纯文本在系统中只作为内部文本中间形态或辅助输入出现，用户不能以“纯文本独立数据域”形式提交、管理或验收该对象。学术图像处理仅承担关键信息提取责任，输出仅限结构化文本、摘要或证据片段，不承担独立图像分析报告生成责任。

MCP 在系统中的角色被限定为按需调用的能力接入层。系统在需要外部工具支持时调用 MCP，不以 MCP 主导业务流程，也不将其写成架构中心特征。本文中的可判断可观测范围仅限每次回答的 token 消耗记录与展示接口预留，不扩展为完整日志、指标或分布式追踪体系。对于具体桌面端框架、向量库、OCR 组件、模型供应商和 MCP 接入方式，本章统一标记为 **待确认**。这些实现型技术细节将在后续技术选型章节中进行比较和裁定，但不会改变本章已经确定的系统边界。
